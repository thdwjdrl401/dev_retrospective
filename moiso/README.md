# 경상북도 모바일 도민증 모이소

## 프로젝트 개요

| 항목 | 내용 |
|---|---|
| 기간 | 2024.04 ~ 2025.08 |
| 역할 | 팀(3인) — 기획·개발·운영 전담 |
| 스택 | Java 11/21, Spring Boot 2~3, JPA, Flutter/Dart, MySQL, GitLab CI/CD |
| 커밋 | 503+ (본인 기준) |
| 규모 | 약 20만 사용자, 8개 이상의 복지 서비스로 구성된 플랫폼 |

경상북도청 발주 복지 통합 플랫폼. 기존 담당자가 구축한 시스템을 이어받아, 합류 후 점차 담당 범위를 넓혀 관리자 웹(`gb_mydata_web`), API 서버(`gb-mydata-api`), Flutter 앱(`gb_mydata`) 세 레포를 모두 담당하게 됐다. 플랫폼 전반의 운영과 모든 서비스의 고도화를 맡았으며, 그중 행복선생님·통합돌봄은 화면 기획서부터 DB 설계, 구현, 배포까지 단독으로 수행했다.

## 내가 풀었던 문제들

### 1. JPA dirty checking으로 복호화한 값이 DB에 다시 저장됐다

개인정보(주민번호, 전화번호)를 조회할 때 DB에서 꺼낸 뒤 복호화해서 반환했다. 처음엔 그냥 `entity.setPhone(decrypt(entity.getPhone()))` 후 반환했는데, 운영 중 개인정보가 평문으로 DB에 덮어써지는 버그가 발생했다.

**원인 추적**: 코드 어디에도 UPDATE 로직이 없어서 원인을 찾기 어려웠다. 추적하던 중 Hibernate 쿼리 로그에서 실제로 UPDATE 문이 나가는 것을 확인했고, 영속성 컨텍스트에 연결된 엔티티의 필드를 바꾸면 트랜잭션 커밋 시점에 dirty checking이 발동해 복호화된 값이 그대로 UPDATE된다는 것을 알게 됐다. 더티 체킹의 동작을 이 과정에서 직접 확인했다.

**해결**: 복호화 전에 `entityManager.detach(entity)`로 영속성 컨텍스트에서 분리한 뒤 필드를 바꾸도록 패턴을 정립했다. 이후 암호화된 필드를 다루는 모든 조회 메서드에 detach를 필수 절차로 적용했다.

---

### 2. 첨부파일 암호화 중 OOM 발생

첨부파일 암호화를 처음엔 파일 전체를 Base64로 메모리에 올린 뒤 처리했다. 소용량에서는 문제없었지만 대용량 파일에서 OOM이 터졌다.

**해결**: `CipherOutputStream` + 8KB 버퍼 스트리밍 방식으로 전환했다. 파일을 청크 단위로 읽어 암호화하면서 바로 출력 스트림에 쓰는 구조라 메모리를 거의 쓰지 않는다. 이후로 파일 처리가 나오면 스트리밍을 기본으로 먼저 고려하게 됐다.

파일 구조도 `[4B 매직패킷 0xDEADBEEF] [16B 랜덤 IV] [AES/CBC 암호문]`으로 정의했다. 헤더 4바이트를 읽어 암호화 여부를 O(1)로 판별할 수 있어, 운영 중 암호화 전·후 파일이 혼재하는 상황에서도 별도 DB 컬럼 없이 처리할 수 있었다.

---

### 3. 행복선생님 QR 출퇴근 — 위경도 중복 문제

QR 코드 스캔 시 경로당을 위경도로 조회하는 구조였는데, 동일 위경도로 등록된 경로당이 존재해 엉뚱한 곳으로 조회되는 버그가 발생했다.

**해결**: QR 데이터에 `qrcodeIdx`(QR코드 고유 식별자)를 포함시키고, 이를 기본 조회 키로 변경했다. 위경도는 보조 수단으로만 남겼다. 공공기관 데이터는 입력 품질을 보장할 수 없다는 걸 다시 한 번 확인한 케이스였다.

---

### 4. 행복선생님 현장 검증 — GPS 권한에 따라 정확도가 달랐다

행복선생님 출퇴근·활동은 현장에 비치된 QR을 GPS 허용 범위 안에서 인식해야 인정되는 구조였다(검증 방식 자체는 기획 단계에서 협의로 정해졌다). 그런데 이를 그대로 구현하면 문제가 있었다. GPS는 사용자가 부여한 위치 권한이 정밀 위치인지 대략 위치인지에 따라 정확도가 크게 달라져, 같은 허용 반경을 적용하면 정상 이용자가 막히거나 반대로 현장에 가지 않고도 인정될 수 있었다.

**해결**: 안드로이드·iOS 공식 문서를 확인해 권한별 정확도 차이를 파악한 뒤, 정밀 위치 제공 여부를 확인하는 로직을 넣었다. 직접 테스트를 반복해 오차 범위를 측정하고 그에 맞춰 허용 반경을 정했다. 또 화면을 다시 촬영한 캡처 이미지로 우회하는 것을 막기 위해 앱 내 카메라의 실시간 인식만 인정하도록 처리했다.

가능한 모든 부정을 다 막기보다, 비용과 효용을 따져 적정 검증 수준을 협의해 정하고 그 안에서 확실히 동작하게 만드는 쪽을 택했다.

### 5. 농어민수당 거주이력 검증 로직이 틀렸다

기존 로직은 "마지막 전입일부터 현재까지 1년 이상이면 통과"였다. 그런데 경북에 살다가 타 지역으로 이사한 뒤 다시 돌아온 사람도 통과시키는 문제가 있었다.

**해결**: 마이데이터 XML의 주소이력을 역순으로 순회하면서, 최근 1년 내 이력 중 경북이 아닌 주소가 하나라도 나오면 즉시 탈락시키는 방식으로 변경했다. "현재 주소 기준"에서 "실제 거주 이력 전체 기준"으로 검증 기준 자체를 바꾼 것이다.

공공 마이데이터 XML은 태그 누락, 빈 값, 인코딩 오류가 빈번하다. 모든 파싱 지점에 null·empty 체크와 예외 처리를 넣는 것이 선택이 아니라 필수임을 이 과정에서 몸으로 익혔다.

### 6. 근무상황표 조회 N+1

행복선생님 근무상황표는 날짜 범위 × 선생님 수만큼 출퇴근 로그를 조회하는 구조였다. 선생님이 많을수록 쿼리가 폭발적으로 늘었고, 한 달 범위 조회 시 응답이 10초를 넘겼다.

**해결**: 전체 로그를 한 번에 로드한 뒤 `Map<teacherIdx, Map<날짜, Map<type, Log>>>` 중첩 구조로 인덱싱해서 루프 안에서 O(1) 접근하도록 변경했다. 페이지네이션도 (날짜, teacherIdx) 키만 먼저 페이징 조회 후 IN 조건으로 상세 데이터를 한 번에 가져오는 2단계 쿼리로 분리했다. 그 결과 같은 조건에서 응답이 0.5초 이내로 줄었다.

원인이 인덱스가 아니라 "전부 가져와 후처리하는 구조"에 있다고 본 것이 핵심이었다. 가져오는 데이터량 자체가 줄지 않으면 단일 쿼리를 빠르게 해도 근본 해결이 되지 않았다.

---

### 7. 엑셀 다운로드 응답 지연 — 사용자가 완료 여부를 모른다

농어민수당 수천 건 엑셀을 생성하면 수십 초가 걸리는데, 그동안 사용자는 아무것도 알 수 없는 상태였다.

**해결**: 서버 인메모리 `HashMap`에 진행률을 저장하고, 클라이언트가 주기적으로 polling해서 진행 상태를 표시하는 방식으로 구현했다. 완료 시 `"완료|{downloadUrl}"` 형태로 반환해 자동 다운로드로 이어지게 했다.

이후 이 방식의 한계를 인식했다. 단일 서버에서는 동작하지만, 다중 서버 환경에서는 진행률 조회 요청이 다른 서버로 라우팅되면 HashMap을 찾을 수 없다. 운영 규모가 커지면 Redis 같은 외부 저장소로 이전해야 한다.

### 8. RSA 키를 매 요청마다 생성하는 비효율

주민등록번호 RSA 암호화 초기 구현에서 요청마다 키쌍을 생성하고 있었다. RSA 키 생성은 무거운 연산이라 성능에 영향을 주었다.

**해결**: `@PostConstruct`로 서버 시작 시 1회만 키쌍을 생성하는 싱글톤으로 전환했다. 키를 파일로 저장하지 않아 서버 재시작 시 자동으로 키가 갱신되는 특성도 자연스럽게 유지됐다.

---

### 9. 처리 완료된 신청 데이터에 주민등록번호가 남아있었다

연간 보안 감리 과정에서, 신청 처리 목적으로 신청 테이블에 임시 저장한 주민등록번호(`userRrn`)가 처리 완료 후에도 DB에 그대로 남아있는 것을 확인했다. 임업인·아픈아이돌봄·교복지원 등 여러 서비스에 걸쳐 동일한 패턴이 있었다. 별도로 `happy_qrcode_log` 테이블에도 주민번호 컬럼이 불필요하게 존재하고 있었다.

**해결**: 각 서비스 처리 완료 시점에 `setUserRrn("")`로 즉시 제거하는 로직을 일괄 추가했다. `happy_qrcode_log`의 `user_rrn` 컬럼은 테이블에서 삭제했다. 또 `User` 객체에 `transient` 필드(`maskRrn`)를 추가해 화면 표시용 마스킹 값을 주민번호 원문과 분리했다.

개인정보는 목적이 달성된 즉시 제거하는 것이 원칙인데, 처음 설계 시 이 원칙을 처리 흐름에 명시적으로 반영하지 않았던 게 문제였다.

---

### 10. Flutter: CI 값이 화면 이동 후 null로 초기화됐다

홈 화면에서 다른 화면으로 이동했다가 돌아오면 CI(Connect Identity) 값이 null이 되는 버그. 원인은 화면 이동 후 세션의 `user` 객체가 재초기화되면서 CI가 사라지는 것이었다.

**해결**: 세션의 `user` 객체 대신 DB를 직접 재조회(`dbUser`)하도록 변경했다. Flutter에서 상태 관리 객체의 생명주기를 충분히 파악하지 못한 채 의존했던 게 원인이었다.

## 설계·판단의 이유

### 매직패킷으로 암호화 여부를 판별한 이유

암호화 적용 시점에 운영 중인 파일이 이미 수천 건 존재했다. 암호화 전 파일과 후 파일이 혼재하는 전환 기간 동안 각각 다르게 처리해야 했다.

선택지는 두 가지였다. DB에 `is_encrypted` 컬럼을 추가하거나, 파일 자체에 상태를 포함시키거나. DB 컬럼 방식은 기존 수천 건 데이터를 일괄 마이그레이션해야 하고, 파일과 DB 상태가 어긋날 가능성도 있었다. 파일 헤더에 4바이트 매직패킷을 박는 방식은 스키마 변경 없이 파일만 보면 상태를 알 수 있고, DB와 파일 상태가 불일치하는 경우 자체가 생기지 않는다. 보안상으로도 파일을 직접 열어보지 않으면 암호화 여부를 알 수 없어 유리하다.

### 행복선생님 권한을 DB 레벨에서 처리한 이유

시군 단위로 접근 범위를 제한하는 권한 체계가 필요했다. 애플리케이션 레이어에서 필터링하면 전체 데이터를 먼저 가져온 뒤 걸러내는 구조가 되고, 데이터가 늘수록 불필요한 로드가 커진다.

`area_code` 컬럼을 `happy_teacher`, `happy_plan`, `happy_plan_result` 세 테이블에 추가해 쿼리 조건 자체에 권한 필터를 포함시켰다. 권한 오류로 인한 데이터 노출 가능성도 DB 레벨에서 원천 차단할 수 있었다.

## 잘 됐던 것

**단독 운영·개발 경험이 밀도 높았다.** 기획, 설계, 구현, 배포, 감리 대응, 유지보수를 혼자 다 했다. 코드의 모든 부분에 맥락이 있었고, 장애가 나면 원인부터 수정까지 직접 추적할 수 있었다. 다른 사람 코드를 건드릴 때의 불확실성 없이 전체를 파악하고 있다는 게 큰 강점이었다.

**감리 대응을 무중단으로 해냈다.** 연간 보안 취약점 진단 조치(EndPoint 권한, XSS, 암호화, 파일 권한 인증)를 운영 중인 서비스에 적용했다. 변경 범위를 최소화하고 사이드 이펙트를 먼저 파악하는 습관이 이때 생겼다.

**행복선생님·통합돌봄을 처음부터 끝까지 만들었다.** 담당 공무원과 요구사항을 정리하고 화면 기획서를 직접 작성했다. 코드만 짜는 게 아니라 "왜 이 기능이 필요한지"를 먼저 이해하고 설계하는 경험이었다.

## 아쉬운 것 / 다음엔 다르게

**엑셀 진행률을 처음부터 Redis로 했어야 했다.** HashMap 방식은 구현이 빠르지만 구조적 한계가 명확하다. 단일 서버 환경이었기 때문에 문제가 없었을 뿐이고, 서버가 늘어나는 순간 무너진다. 처음부터 외부 저장소를 쓰는 게 맞았다.

**파일 처리에서 OOM을 겪고 나서야 스트리밍을 배웠다.** Base64 전체 로드가 문제가 될 거라는 걸 설계 시점에 미리 알았어야 한다. 파일을 메모리에 올리는 코드가 나오면 먼저 크기 상한을 확인하는 습관이 생겼지만, 순서가 반대였다.

**공공 데이터의 품질 가정을 너무 낙관했다.** 마이데이터 XML 파싱에서 초반에 null 체크를 일부 빠뜨렸다가 운영 중 NPE가 발생했다. 외부 시스템 데이터는 어떤 경우에도 방어적으로 처리해야 한다는 원칙을 이때 확립했다.

**Spring Boot 2 → 3 마이그레이션을 미뤘다.** 레포마다 버전이 달라 관리 비용이 생겼다. 이어받을 때 버전 통일을 먼저 검토했어야 했다.
