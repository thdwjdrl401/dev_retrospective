# 중고차 부품 거래 플랫폼 카썹

## 프로젝트 개요

| 항목 | 내용 |
|---|---|
| 기간 | 2025.08 ~ 2025.09 |
| 역할 | 팀 참여 — JWT 인증 적용, 리뷰/신고 기능 신규 개발 |
| 스택 | Spring Boot 3.2.0, Java 21, JPA / Flutter, Dart |
| 커밋 | 11 (본인 기준, 전체 770+) |

고객이 폐차장의 중고부품을 직접 구매하고, 정비소까지 연결하는 플랫폼. 팀에서 이미 개발 중이던 프로젝트에 합류해 인증이 없던 앱 API 전체에 JWT를 붙이고, 리뷰 신고 기능을 신규 설계·구현했다.

---

## 내가 풀었던 문제들

### 1. 인증 없이 열려있던 앱 API 전체

기존에 앱에서 호출하는 API EndPoint 전체가 인증 없이 접근 가능한 상태였다. JWT 인증 유틸 클래스를 직접 설계하고, 47개 파일에 인증 로직을 일괄 적용했다.

토큰 검증 순서를 단계별로 명확히 구분했다. Bearer 토큰 존재 여부 → JWT 형식 검사 → DB 사용자 조회 → 서명·만료·Subject 일치 검증 순서로 흘러가고, 각 단계에서 실패 원인에 따라 401/403/404를 구분해 반환하도록 설계했다. 컨트롤러마다 반복되는 인증 로직은 `authenticateAndAuthorize()` 한 메서드로 캡슐화해 중복을 없앴다.

미로그인 사용자도 조회 전용 EndPoint는 쓸 수 있어야 해서, Guest 토큰을 별도 발급하는 로직도 함께 추가했다.

---

### 2. 컨트롤러마다 흩어진 세션 권한 검증

각 컨트롤러가 `HttpSession`을 파라미터로 직접 받아 세션을 꺼내는 코드가 중복되어 있었다. 권한 검증 로직이 각자 다르게 흩어져 있으면 이후 정책 변경 시 전부 수정해야 한다.

`PermissionChecker`라는 공통 컴포넌트를 만들어 권한 검증을 한 곳으로 모았다. 싱글톤 빈에서 request-scoped 빈에 접근해야 하는 문제는 `Provider<SessionUser>`로 주입해 해결했다. `RequestContextHolder`를 써서 `HttpServletRequest`를 파라미터 없이 접근할 수 있게 했고, 미인가 시 현재 URL을 `?url=` 파라미터로 보존해 로그인 후 원래 페이지로 복귀하도록 처리했다.

돌아보면 Spring Security의 `@PreAuthorize`나 `SecurityConfig`의 path 기반 설정으로 Security 레이어에서 처리하는 게 더 나은 선택이었다. 직접 만든 방식은 권한 정책이 여전히 컨트롤러 코드에 묻혀 있고, `isRole()`에서 `IOException`을 던지는 것도 권한 검증과 HTTP 처리가 섞인 설계였다.

---

### 3. 신고된 리뷰가 삭제되면 내용을 확인할 수 없다

리뷰 신고 기능을 설계할 때, 신고된 리뷰가 이후 삭제되면 관리자가 신고 내용을 확인할 수 없는 문제를 미리 고려했다.

Report 엔티티에 `original` 필드를 추가해 신고 시점의 원본 내용을 스냅샷으로 저장하도록 설계했다. 원본이 삭제되거나 수정돼도 신고 당시 상태를 항상 확인할 수 있다.

---

### 4. Flutter Confetti 애니메이션이 기기마다 다르게 동작했다

결제 완료 화면의 Confetti 애니메이션이 기기별 디스플레이 주사율에 따라 불규칙하게 동작하는 버그가 있었다. 원인은 `stop()` 호출을 고정 ms 타이머에 의존하고 있어서, 주사율이 다른 기기에서는 타이머가 프레임 경계와 어긋났다.

`SchedulerBinding.addPostFrameCallback()`으로 Flutter 엔진이 현재 프레임을 완전히 렌더링한 직후에 `stop()`을 호출하도록 변경했다. 기기 주사율과 무관하게 정확히 1프레임 후에 실행되므로 기기별 차이가 없어졌다.

---

## 설계·판단의 이유

### Report original 스냅샷을 엔티티에 직접 포함한 이유

신고 내용 보존 방법으로 원본 테이블에 `is_reported` 플래그를 두고 삭제를 막는 방식도 있었다. 하지만 그 방식은 신고 여부가 원본 데이터의 생명주기를 제어하게 되어 두 도메인이 결합된다. Report 엔티티 안에 스냅샷을 저장하면 신고 도메인이 독립적으로 원본 내용을 보존할 수 있고, 원본 삭제 정책을 신고 로직과 분리해서 다룰 수 있다.

## 잘 됐던 것

**인증 로직 공통화가 실질적 효과가 있었다.** 47개 EndPoint에 인증을 붙이는 작업을 메서드 하나 추가로 처리할 수 있었다. 공통화 시점에 조금 더 들인 설계 비용이 적용 단계에서 회수됐다.

**Flutter 프레임레이트 버그 원인을 정확히 잡았다.** ms 기반 타이머가 아니라 Flutter 렌더링 파이프라인의 콜백 시점 차이라는 걸 파악한 게 핵심이었다.

---

## 아쉬운 것 / 다음엔 다르게

**PermissionChecker를 직접 만드는 대신 Spring Security를 썼어야 했다.** 직접 만든 방식은 권한 정책이 컨트롤러 곳곳에 퍼져있는 구조를 그대로 가져간다. Security 레이어에서 처리하면 권한 정책이 한 곳에 집중되고, 이미 검증된 방식을 쓰는 게 맞다. 이미 구축된 프로젝트에 합류할 때는 기존 설계 흐름을 먼저 파악하고 그 위에서 작업하는 게 중요하다는 걸 다시 확인했다.
