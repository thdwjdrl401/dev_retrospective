# 입시 상담 예약 서비스 tomas

## 프로젝트 개요

| 항목 | 내용 |
|---|---|
| 기간 | 2024.05 ~ 2024.09 |
| 역할 | 팀(3인) — 관리자 CRUD, 공지사항, TinyMCE, 생기부 첨부파일, GTM |
| 스택 | Spring Boot 3.2.0, Java 21, Thymeleaf, MySQL |
| 커밋 | 65 (본인 기준, 전체 717 / main 반영 기준) |

학원 입시 상담 예약 관리 시스템. 학생·학부모가 상담을 예약하고, 상담사·분석가·관리자가 각 역할에 맞게 처리하는 구조다. 입사 초기에 맡은 프로젝트로, 기능이 대체로 구현된 상태에서 관리자 기능과 콘텐츠 관리 영역의 마무리·운영을 담당했다. 실무에서 Spring과 웹 개발의 기본기를 익혀 나가던 시기의 작업이다.

---

## 이때 다뤘던 것들

### 1. 비밀번호 검증이 평문 비교로 되어 있었다

비밀번호 변경 모달을 작업하다, 현재 비밀번호 검증 로직이 평문으로 비교하고 있는 것을 발견했다. DB에는 BCrypt로 암호화된 값이 저장되어 있어 평문 비교는 정상 동작할 수 없는 상태였다.

`BCryptPasswordEncoder.matches()`로 수정했다. 당시에는 암호화 저장과 검증 방식이 아직 익숙하지 않았는데, 이 버그를 고치면서 비밀번호는 단순 비교가 아니라 인코더를 통해 검증해야 한다는 걸 실제 코드로 익혔다.

---

### 2. 공지사항 순서를 관리자가 직접 바꿀 수 없었다

공지사항을 관리자가 원하는 순서로 조정해야 하는 요구사항이 있었다. 등록일 정렬만으로는 세밀한 순서 제어가 어려웠다.

`placeOrder` 필드를 두고 드래그 앤 드롭으로 순서를 바꾸도록 구현했다. 공지 고정(`isNotice`)은 별도로 두어, 고정 공지는 상단에 두고 그 아래는 `placeOrder` 순으로 정렬했다. 정렬 기준을 데이터로 관리하는 방식을 이때 처음 다뤄봤다.

---

### 3. TinyMCE 에디터를 기존 폼에 통합했다

메뉴얼 콘텐츠를 일반 `textarea`로 받으면 서식 없는 평문만 저장된다. 관리자가 강조·링크·이미지를 포함한 콘텐츠를 작성할 수 있어야 했다.

TinyMCE WYSIWYG 에디터를 기존 Thymeleaf 폼에 통합하고, 한국어 언어팩(`ko_KR.js`)을 적용했다. 에디터에서 작성한 HTML이 `ManualController`를 통해 저장·조회되도록 연결했다. 외부 라이브러리를 기존 화면에 붙여 동작시키는 과정을 경험했다.

---

## 이때 배운 것

**기존 코드의 보안 버그를 그냥 지나치지 않았다.** 평문 비교 버그는 내가 만든 코드가 아니었지만, 맡은 모달을 작업하다 발견해 고쳤다. 초기였지만 보안과 관련된 부분은 작아도 그냥 넘기면 안 된다는 감각을 이때 얻었다.

**맡은 범위를 기간 안에 완결하는 경험을 했다.** 담당한 관리자 기능과 콘텐츠 관리 영역을 끝까지 마무리해 넘겼다. 팀 안에서 자기 몫을 책임지고 완결하는 게 어떤 것인지 처음 체감한 프로젝트였다.

---

## 돌아보면

입사 초기, Spring과 웹 개발의 기본기를 익혀 나가던 시기의 작업이다. 당시에는 맡은 기능 중심으로 빠르게 처리하느라 전체 도메인 구조를 충분히 파악하지 못한 채 작업한 부분도 있었다. 이때의 경험이 이후 모이소처럼 더 큰 서비스를 단독으로 맡을 때, 기능보다 전체 흐름을 먼저 파악하는 습관으로 이어졌다.
