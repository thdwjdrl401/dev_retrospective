# 덕성여대 산학협력단 홈페이지

## 프로젝트 개요

| 항목 | 내용 |
|---|---|
| 기간 | 2024.03 ~ 2025.09 |
| 역할 | 메인 담당 (웹 관리자 백엔드·프론트 전체) |
| 스택 | Java, Spring Boot, JPA, Thymeleaf, MySQL, GitLab CI/CD |
| 커밋 | 약 90 (본인 기준, 전체 808) |

덕성여자대학교 산학협력단이 운영하는 산학협력 실적·RnD 성과 통합 관리 시스템. 교수·연구자가 산학협력 실적을 등록하면 관리자가 심사·승인하고, 외부 HRM 시스템과 연계하는 구조. 기존 담당자가 구축한 시스템에 합류해, 초기에는 다른 개발자들과 함께 작업하다 점차 메인 담당을 맡아 약 1.5년간 유지보수·개발했다.

---

## 내가 풀었던 문제들

### 1. 카테고리마다 HRM 연계 코드값과 필드 구조가 달랐다

산학협력 실적이 최종 승인되면 외부 HRM 시스템의 뷰 테이블(`vw_rrm_coopr_achv`)에 데이터를 적재해야 했다. 문제는 HRM 시스템이 기대하는 코드값과 CONT 필드 구조가 카테고리(지식재산권·기술이전·산학협력)마다 전부 달랐다는 것이다.

지식재산권의 경우 입력 항목이 최대 14개인데, 이를 `type3` 필드에 `@@` 구분자로 직렬화한 뒤 파싱해서 `CONT_1~15`에 각각 매핑한다. 여기서 각 값은 단순 텍스트가 아니라 HRM 코드값(`RRM019.10`, `RRM019.20` 등)으로 변환해야 해서, `common` 테이블에 JSON 형태의 코드 매핑 테이블(`CONT_CODE_CONVERT`)을 두고 런타임에 조회하는 방식을 설계했다. 코드값이 바뀌더라도 소스코드 수정 없이 DB에서 관리할 수 있게 하기 위해서였다.

산학협력 카테고리는 `BusinessCategory.subContent` 필드에 JSON으로 저장된 구성 정보를 읽어 어떤 항목이 활성화됐는지 동적으로 결정하고, 그에 따라 CONT 필드를 매핑했다. 카테고리마다 구조가 다른 문제를 코드 분기가 아니라 데이터로 풀어낸 셈이다.

---

### 2. 관리자와 교수에게 보여야 하는 상태가 달랐다

심사 상태(`status2`)가 100·200·300 세 단계로 나뉘는데, 관리자는 단계별로 상세히 파악해야 하지만 교수에게는 진행 중인 모든 단계를 "심사중" 하나로 보여야 혼란이 없었다. 관리자용 `status2Text()`와 사용자용 `status2TextToUser()`를 별도로 두어 같은 데이터를 목적에 맞게 다르게 노출했다.

신청 상태(`status`)와 심사 상태(`status2`)가 독립적으로 움직이는 구조였다. 두 상태의 조합에 따라 가능한 액션이 달라지는데, 이를 뷰 레이어에서 분기 처리하는 대신 도메인 메서드(`statusCode()`, `status2Code()`)에 코드값 변환 로직을 캡슐화해 컨트롤러·뷰가 상태 체계를 직접 다루지 않도록 했다.

---

### 3. 교수가 탈퇴하면 실적 기록에서 이름이 사라졌다

기존에는 실적 목록·상세에서 신청자 이름을 `user` 테이블 조인으로 가져왔다. 교수가 탈퇴하면 해당 실적에서 신청자 이름이 사라지는 문제가 있었다.

`businessresult` 테이블에 `userName` 컬럼을 추가해 등록·수정 시점의 이름을 스냅샷으로 저장하도록 변경했다. 이후 이름 표시는 조인 대신 `userName` 컬럼에서 직접 읽는다. 원본이 변경·삭제돼도 이력 데이터는 당시 상태를 보존해야 한다는 원칙이다.

---

### 4. URL idx 파라미터를 변조해 다른 사람 데이터에 접근할 수 있었다

합류 직후 URL의 `idx` 파라미터를 임의로 바꾸면 다른 교수의 실적 상세, 공용장비 신청 내역 등에 접근할 수 있는 취약점을 다수 발견했다.

컨트롤러에서 세션 사용자와 데이터 소유자를 대조하는 검증을 일괄 추가했다. 불일치 시 홈으로 리다이렉트. 인증과 인가는 다른 문제라는 걸 다시 확인했다 — 로그인했다고 해서 모든 데이터에 접근할 수 있어야 한다는 뜻은 아니다.

---

### 5. 지식재산권 type3 파싱 — 기존 직렬화 구조를 그대로 이어받았다

지식재산권 입력 항목 14개가 `@@` 구분자로 직렬화된 `type3` 컬럼에 저장되어 있었다. 정규화 관점에서는 별도 테이블이 맞지만, 기존 시스템 전반이 이 구조를 전제하고 있어 이어받은 시점에 변경하기 어려웠다.

대신 파싱 로직을 서비스 레이어에 집중시켜, 컨트롤러·뷰는 `type3`의 직렬화 구조를 직접 다루지 않도록 캡슐화했다. 기술적 부채를 당장 해소하지 못하더라도 영향 범위를 최소화하는 선택이었다.

---

### 6. 윈도우 서버가 강제 업데이트로 반복해서 다운됐다

운영 중인 윈도우 서버가 강제 업데이트 후 재부팅되면서 서비스가 다운되는 일이 주기적으로 발생했다. 주변에서는 어쩔 수 없는 일로 여겼지만, 최소한 다운을 먼저 인지하는 것은 필요하다고 판단했다.

지시받은 DB 백업 작업에 더해, 서버 상태를 점검하고 다운 시 알리는 모니터링·알림 체계를 자발적으로 구축했다. SSH로 접속해 `mysqldump`로 여러 DB를 정기 백업하고 외부 저장소에 보관했으며, health check로 서버 상태를 주기적으로 확인해 다운이 감지되면 메일로 알리고 대시보드에서 상태를 볼 수 있게 했다. 그 결과 고객이 알리기 전에 장애를 먼저 감지한 경우가 여러 번 있었다.

다만 이 방식은 다운을 빨리 아는 것이지 막는 것은 아니었다. 근본 원인을 해소하기 위해, 최종적으로 리눅스 서버로 교체하는 방향으로 정리됐다.

## 설계·판단의 이유

### 코드 매핑을 DB JSON으로 관리한 이유

HRM 코드값은 외부 시스템의 정책에 따라 언제든 바뀔 수 있다. 코드값을 소스코드에 하드코딩하면 변경마다 배포가 필요하지만, `common` 테이블에 JSON으로 저장하면 운영 중에도 DB 수정만으로 반영할 수 있다. 외부 시스템 연계에서 변동 가능한 값은 코드가 아니라 데이터로 관리하는 게 유지보수 비용을 낮춘다.

### 상태 표시를 도메인 메서드로 캡슐화한 이유

status 코드값(`RRM030.20`, `RRM030.30` 등)이 뷰·컨트롤러 곳곳에 흩어지면 코드값 변경 시 수정 범위를 파악하기 어렵다. `statusCode()`, `status2Code()` 메서드로 변환 로직을 도메인에 모으면 코드값이 바뀌어도 한 곳만 수정하면 된다.

## 잘 됐던 것

**외부 시스템 연계 구조를 직접 설계했다.** 코드값 매핑, 카테고리별 CONT 필드 분기, 동적 매핑까지 설계부터 구현까지 직접 맡았다. 외부 스펙이 고정된 상황에서 내부 설계를 얼마나 유연하게 가져갈 수 있는지 고민했다.

**운영 안정성을 위해 먼저 움직였다.** 서버 다운을 당연하게 여기지 않고, 지시받은 백업 범위를 넘어 모니터링·알림 체계를 자발적으로 만들었다. 임시 대응(빨리 감지)과 근본 해결(서버 교체)을 구분해서 접근했다.

**1.5년간 이어받은 시스템의 맥락을 파악하며 유지보수했다.** 기존 코드 전체를 파악하지 않으면 수정이 어려운 구조였다. 변경 전 영향 범위를 먼저 확인하는 습관이 이 과정에서 굳어졌다.

## 아쉬운 것 / 다음엔 다르게

**type3 직렬화 구조를 초기에 리팩터링했어야 했다.** 이어받을 때 기존 구조를 그대로 유지하는 게 안전하다고 판단했지만, 이 구조는 이후 CONT 매핑 로직이 복잡해질수록 파싱 코드가 함께 늘어나는 문제를 만들었다. 이어받은 직후가 리팩터링 비용이 가장 낮은 시점이었다.

**상태 체계 문서화를 코드 외부에 남기지 않았다.** `status`와 `status2`의 조합 가능 경우의 수가 많아서, 처음 파악할 때 코드를 역으로 추적해야 했다. 복잡한 상태 체계는 코드 주석이나 별도 문서로 전체 매트릭스를 기록해두는 게 인수인계 비용을 낮춘다.

**모니터링 구축 시 SSH 직접 접근에 의존했다.** 빠르게 만들기 위해 운영 서버에 SSH로 직접 접속하는 방식을 썼는데, 보안 관점에서는 위험이 있었다. 지금이라면 서버에 상태를 노출하는 별도 경로를 두고 그쪽을 점검하는 방식으로 접근했을 것이다.
