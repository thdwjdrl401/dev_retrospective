# asnet 내부 운영 대시보드

## 프로젝트 개요

| 항목 | 내용 |
|---|---|
| 기간 | 2024.07 |
| 역할 | 단독 — 전체 설계·구현 |
| 스택 | Spring Boot, Java, JSch, Google Drive API, JavaMail, Apache Tika |
| 커밋 | 12 (전체 동일) |

회사 내부 운영 도구. 여러 서버에 분산된 DB를 자동으로 백업해 Google Drive에 업로드하고, 운영 중인 9개 사이트의 상태를 주기적으로 점검해 장애 발생 시 이메일로 알리는 시스템이다. 지시받은 게 아니라, 반복적인 수동 백업과 서버 다운 인지 지연 문제를 직접 해결하기 위해 만들었다.

---

## 내가 풀었던 문제들

### 1. 여러 서버의 DB를 수동으로 백업하고 있었다

관리 중인 프로젝트 DB가 r1, r3, r5, tomas 등 여러 서버에 흩어져 있었다. DB 포트를 외부에 직접 열지 않고 백업하려면 서버에 접속해서 `mysqldump`를 실행해야 했다.

JSch 라이브러리로 SSH 터널을 통해 원격 서버에 접속하고, `echo {password} | sudo -S mysqldump ...` 명령을 실행해 SQL 덤프를 스트림으로 읽어 로컬에 파일로 저장한 뒤, Google Drive에 업로드하고 로컬 파일을 삭제하는 흐름으로 구현했다. DB 포트 개방 없이 SSH 권한만으로 백업이 가능하다.

서버별 SSH/DB 크리덴셜은 `DBInfo` 모델에 캡슐화하고, `setR1DB()`, `setTomasDB()` 등 서버별 팩토리 메서드로 세팅해 백업 로직이 크리덴셜 상세를 알지 않아도 되게 했다.

---

### 2. 백업 주기를 서버마다 다르게 가져가야 했다

tomas처럼 활성 운영 중인 서비스는 매일 백업이 필요하지만, 변경이 적은 서비스(newlog, lincplus, diversitas 등)는 매일 백업하면 Drive 저장 공간이 불필요하게 소모된다.

`@Scheduled(cron = "0 0 4 * * ?")`로 매일 새벽 4시에 tomas를 백업하고, 같은 스케줄러 안에서 월말(`isLastDayOfMonth()`) 여부를 확인해 나머지 서비스는 월 1회만 실행하도록 분기했다. 스케줄러를 여러 개 만드는 대신 하나의 흐름 안에서 조건으로 관리한 셈이다.

---

### 3. 사이트가 다운돼도 한참 뒤에야 알게 됐다

운영 중인 9개 사이트(덕성여대 산학, 동국대 pshare, tomasedu 등)가 다운되면 고객이 먼저 연락하거나 우연히 접속해봐야 알 수 있었다.

10초마다 각 사이트에 HTTP GET 요청을 보내 응답 코드를 확인하고, 상태가 바뀐 경우에만 이메일을 발송하도록 했다. `previousStatus` 맵으로 직전 상태를 기록해, 다운 상태가 지속되는 동안 10초마다 이메일이 쏟아지는 문제를 방지했다. 장애 감지 시 개인 메일과 회사 메일 양쪽으로 동시 발송한다.

---

### 4. Google Drive 업로드 시 MIME 타입을 알 수 없었다

Drive API는 업로드 시 MIME 타입을 명시해야 한다. SQL 덤프 파일을 하드코딩으로 처리하면 이후 다른 형식이 추가될 때마다 분기가 늘어난다.

Apache Tika로 파일을 분석해 MIME 타입을 자동 감지하도록 했다. 파일 형식이 바뀌어도 코드 수정 없이 처리된다.

---

## 설계·판단의 이유

### SSH 터널을 통해 mysqldump를 실행한 이유

DB 포트(3306)를 외부에 여는 것은 보안상 피해야 했다. SSH 포트(22)는 이미 열려있고 접근 제어가 되어 있어, 기존 SSH 권한을 그대로 활용하는 방식이 추가 인프라 변경 없이 안전하게 백업할 수 있는 경로였다.

### 상태 변경 시에만 알림을 보낸 이유

10초 간격으로 체크하면서 매번 알림을 보내면 다운 상태가 지속되는 동안 이메일이 폭증한다. 실제로 필요한 정보는 "언제 다운됐는가"와 "언제 복구됐는가"이지, 매 10초마다의 상태 확인 결과가 아니다. `previousStatus`와 현재 상태를 비교해 전환이 일어날 때만 알리는 방식이 운영 노이즈를 최소화한다.

---

## 잘 됐던 것

**필요하다고 판단해서 먼저 만들었다.** 지시받은 작업이 아니라, 반복 수동 작업과 장애 인지 지연이라는 실제 문제를 보고 직접 도구를 만들었다. 이후 사이트 다운을 고객보다 먼저 감지하는 경우가 생겼다.

**SSH 터널 백업 구조가 실용적으로 동작했다.** 서버 인프라 변경 없이 기존 SSH 접근만으로 여러 서버의 DB를 자동화할 수 있었다.

---

## 아쉬운 것 / 다음엔 다르게

**헬스체크 간격이 10초로 너무 짧았다.** HTTP 요청을 9개 사이트에 10초마다 보내는 건 과하다. 실제 장애 감지에 5분 간격으로도 충분하고, 그쪽이 외부 서비스에 대한 부하도 적다.

**SSH 크리덴셜을 `Const` 클래스에 하드코딩했다.** 빠르게 만드는 데 집중하다 보니 환경변수나 별도 설정 파일로 분리하지 않았다. 운영 도구라도 크리덴셜은 코드 외부에서 주입하는 게 맞다.

**단순 HTTP 200 체크는 서비스 정상 여부를 보장하지 않는다.** 웹 서버가 200을 반환해도 DB 연결이 끊겼거나 내부 오류가 있는 경우를 잡지 못한다. 실제 기능을 확인하는 엔드포인트를 별도로 두는 게 더 정확한 모니터링이다.
